# Building a Plaid Convex component from existing integration

**No Plaid component exists in the Convex ecosystem today**, making this a valuable opportunity. Your existing 11-action, 7-table implementation can be packaged as a reusable component, but requires architectural changes to accommodate Convex's component isolation model. The Stripe component (`@convex-dev/stripe`) provides the best reference pattern for similar webhook-heavy external API integrations.

## Component architecture fundamentals

Convex components are **self-contained backend modules** with their own isolated database tables, file storage, and scheduled functions. They're defined using `defineComponent` in a `convex.config.ts` file rather than `defineApp`:

```typescript
// src/component/convex.config.ts
import { defineComponent } from "convex/server";

const component = defineComponent("plaid");
export default component;
```

Consumers install your component in their app's configuration:

```typescript
// convex/convex.config.ts (consumer's app)
import { defineApp } from "convex/server";
import plaid from "@your-org/convex-plaid/convex.config.js";

const app = defineApp();
app.use(plaid);
// Multiple instances supported with different names
app.use(plaid, { name: "plaidSecondary" });
export default app;
```

The critical architectural differences between regular Convex functions and components include **complete database isolation** (components cannot access app tables and vice versa), **no `ctx.auth` access** (authentication must be handled in the host app with userId passed explicitly), **no `process.env` access** (secrets must be passed as function arguments), and **no direct HTTP route exposure** (the app must mount component webhook handlers).

## Schema design with complete table isolation

Components define their own schema that's **completely isolated** from the host app. Your 7 Plaid tables become component-internal:

```typescript
// src/component/schema.ts
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({

    userId: v.string(),           // Note: v.string(), not v.id("users")

    itemId: v.string(),
    institutionId: v.optional(v.string()),
    institutionName: v.optional(v.string()),
    cursor: v.optional(v.string()),
    consentExpirationTime: v.optional(v.number()),
    lastSyncedAt: v.optional(v.number()),

    .index("by_item_id", ["itemId"]),

  plaidAccounts: defineTable({
    plaidItemId: v.id("plaidItems"),

    name: v.string(),

    transactionId: v.string(),
    amount: v.number(),
    date: v.string(),
    name: v.string(),
    merchantName: v.optional(v.string()),
    category: v.optional(v.array(v.string())),
    categoryId: v.optional(v.string()),
    pending: v.boolean(),
    // Enrichment fields
    enrichedCategory: v.optional(v.string()),
    enrichedMerchant: v.optional(v.string()),
  })
    .index("by_account", ["plaidAccountId"])
    .index("by_transaction_id", ["transactionId"])
    .index("by_date", ["date"]),

  // Similar definitions for liabilities tables...
  plaidCreditCardLiabilities: defineTable({
    plaidAccountId: v.id("plaidAccounts"),
    // ... fields
  }).index("by_account", ["plaidAccountId"]),
  
  plaidRecurringStreams: defineTable({
```

**Critical ID type consideration**: All `Id<"tableName">` types become plain `string` when crossing component boundaries. Your component's public API should accept and return strings for IDs, even though internal functions use typed IDs.

## Configuration and secrets handling

Components **cannot access `process.env`** directly. The recommended pattern uses a class-based client that accepts configuration at instantiation, falling back to environment variables when called from the host app:

```typescript
// src/client/index.ts
import type { ComponentApi } from "../component/_generated/component.js";
import type { GenericActionCtx, GenericMutationCtx, GenericQueryCtx, GenericDataModel } from "convex/server";

export interface PlaidConfig {
  PLAID_CLIENT_ID?: string;
  PLAID_SECRET?: string;
  PLAID_ENV?: "sandbox" | "development" | "production";
  webhookUrl?: string;
  // Encryption key for access tokens
  ENCRYPTION_KEY?: string;
}

export class Plaid {
  private config: Required<PlaidConfig>;

  constructor(
    public component: ComponentApi,
    options?: PlaidConfig
  ) {
    // Fall back to environment variables (read in host app context)
    this.config = {
      PLAID_CLIENT_ID: options?.PLAID_CLIENT_ID ?? process.env.PLAID_CLIENT_ID!,
      PLAID_SECRET: options?.PLAID_SECRET ?? process.env.PLAID_SECRET!,
      PLAID_ENV: options?.PLAID_ENV ?? (process.env.PLAID_ENV as any) ?? "sandbox",
      webhookUrl: options?.webhookUrl ?? process.env.PLAID_WEBHOOK_URL!,
      ENCRYPTION_KEY: options?.ENCRYPTION_KEY ?? process.env.PLAID_ENCRYPTION_KEY!,
    };
  }

  async createLinkToken(
    ctx: GenericActionCtx<GenericDataModel>,
    args: { userId: string; products?: string[] }
  ) {
    return await ctx.runAction(this.component.public.createLinkToken, {
      ...args,
      ...this.config, // Pass all config as args
    });
  }

  async exchangePublicToken(
    ctx: GenericMutationCtx<GenericDataModel>,
    args: { userId: string; publicToken: string; institutionId?: string }
  ) {
    return await ctx.runMutation(this.component.public.exchangePublicToken, {
      ...args,
      ...this.config,
    });
  }

  // Additional methods for all 11 actions...
}
```

The host app instantiates and uses the component:

```typescript
// Consumer's convex/plaid.ts
import { Plaid } from "@your-org/convex-plaid";
import { components } from "./_generated/api";

export const plaid = new Plaid(components.plaid, {
  // Optional overrides; defaults read from process.env
  PLAID_ENV: "production",
});
```

## Authentication integration patterns

Since components lack `ctx.auth` access, authentication **must happen in the host app** with user identifiers passed explicitly. Your existing `requireAuth` and `requireOwnership` patterns need restructuring:

```typescript

export const syncTransactions = action({

    PLAID_SECRET: v.string(),
    PLAID_ENV: v.string(),
    ENCRYPTION_KEY: v.string(),

  returns: v.object({
    added: v.number(),
    modified: v.number(),

    const item = await ctx.runQuery(internal.items.getByItemId, { itemId: args.itemId });
    if (!item || item.userId !== args.userId) {
      throw new ConvexError({ code: "NOT_FOUND", message: "Item not found" });
    }
    
    // Proceed with sync...
  },
});
```

For consumers who want to expose component functionality to clients with auth, provide a factory function:

```typescript
// src/client/index.ts
import { mutation, query, action } from "convex/server";
import { v } from "convex/values";
import type { Auth } from "convex/server";

export function makePlaidAPI(
  component: ComponentApi,
  options: {
    auth: (ctx: { auth: Auth }) => Promise<string | null>;
    config: PlaidConfig;
  }
) {
  return {
    // Exposed to clients with authentication wrapper
    getAccounts: query({
      args: {},
      handler: async (ctx) => {
        const userId = await options.auth(ctx);
        if (!userId) throw new Error("Unauthenticated");
        return await ctx.runQuery(component.public.getAccountsByUser, {
          userId,
        });
      },
    }),

    createLinkToken: action({
      args: { products: v.optional(v.array(v.string())) },
      handler: async (ctx, args) => {
        const userId = await options.auth(ctx);
        if (!userId) throw new Error("Unauthenticated");
        return await ctx.runAction(component.public.createLinkToken, {
          userId,
          products: args.products,
          ...options.config,
        });
      },
    }),
  };
}
```

## Webhook handling architecture

Components **cannot expose HTTP routes directly**—routes must be mounted by the host app. Following the Stripe component pattern, provide a route registration function:

```typescript
// src/client/index.ts
import { httpRouter, httpAction } from "convex/server";
import type { ComponentApi } from "../component/_generated/component.js";

export class Plaid {
  // ... other methods

  registerWebhookRoutes(
    http: ReturnType<typeof httpRouter>,
    options?: {
      path?: string;
      webhookSecret?: string;
      onWebhook?: (ctx: any, webhook: PlaidWebhook) => Promise<void>;
    }
  ) {
    const path = options?.path ?? "/webhooks/plaid";

    http.route({
      path,
      method: "POST",
      handler: httpAction(async (ctx, request) => {
        // Verify webhook signature
        const body = await request.text();
        const signature = request.headers.get("plaid-verification");
        
        const isValid = await ctx.runAction(
          this.component.internal.verifyWebhookSignature,
          {
            body,
            signature,
            webhookSecret: options?.webhookSecret ?? this.config.PLAID_WEBHOOK_SECRET,
          }
        );

        if (!isValid) {
          return new Response("Invalid signature", { status: 401 });
        }

        const webhook = JSON.parse(body) as PlaidWebhook;

        // Process webhook in component
        await ctx.runMutation(this.component.internal.processWebhook, {
          webhook,
          ...this.config,
        });

        // Optional callback for custom handling
        if (options?.onWebhook) {
          await options.onWebhook(ctx, webhook);
        }

        return new Response(null, { status: 200 });
      }),
    });
  }
}
```

Consumer mounts the routes in their `http.ts`:

```typescript
// Consumer's convex/http.ts
import { httpRouter } from "convex/server";
import { plaid } from "./plaid";

const http = httpRouter();

plaid.registerWebhookRoutes(http, {
  path: "/webhooks/plaid",  // Configurable endpoint
  onWebhook: async (ctx, webhook) => {
    // Custom handling: notify user, trigger app-specific logic
    if (webhook.webhook_type === "TRANSACTIONS" && 
        webhook.webhook_code === "SYNC_UPDATES_AVAILABLE") {
      await ctx.runMutation(api.notifications.create, {
        type: "transactions_updated",
        itemId: webhook.item_id,
      });
    }
  },
});

export default http;
```

## Scheduled sync implementation

Components can define static cron jobs in their own `crons.ts`:

```typescript
// src/component/crons.ts
import { cronJobs } from "convex/server";
import { internal } from "./_generated/api";

const crons = cronJobs();

// Daily sync at 2 AM UTC
crons.cron(
  "daily-transaction-sync",
  "0 2 * * *",
  internal.sync.syncAllActiveItems
);

export default crons;
```

For **configurable scheduling**, use the `@convex-dev/crons` component as a child component:

```typescript
// src/component/convex.config.ts
import { defineComponent } from "convex/server";
import crons from "@convex-dev/crons/convex.config.js";

const component = defineComponent("plaid");
component.use(crons);
export default component;
```

Then expose scheduling configuration:

```typescript
// src/client/index.ts
export class Plaid {
  async configureSyncSchedule(
    ctx: GenericMutationCtx<GenericDataModel>,
    args: { cronspec: string }  // e.g., "0 2 * * *"
  ) {
    return await ctx.runMutation(this.component.internal.configureSyncSchedule, {
      cronspec: args.cronspec,
    });
  }
}
```

## Multi-tenant considerations

Your architecture supports multiple Plaid items per user. In the component model, this works naturally since all items are stored in the component's isolated `plaidItems` table with a `userId` field:

```typescript
// Component query: get all items for a user
export const getItemsByUser = query({
  args: { userId: v.string() },
  returns: v.array(/* plaidItem validator */),
  handler: async (ctx, args) => {
    return await ctx.db
      .query("plaidItems")
      .withIndex("by_user", (q) => q.eq("userId", args.userId))
      .filter((q) => q.eq(q.field("status"), "active"))
      .collect();
  },
});
```

For **multi-tenant SaaS** where different organizations use your component, the host app manages tenant isolation. The component receives tenant/user identifiers and stores them:

```typescript
// Enhanced schema for multi-tenant
plaidItems: defineTable({
  tenantId: v.optional(v.string()),  // For SaaS multi-tenancy
  userId: v.string(),
  // ...
}).index("by_tenant_user", ["tenantId", "userId"])
```

## API design recommendations

Based on patterns from successful Convex components, your Plaid component's public API should follow these conventions:

**Consistent method signatures** matching your existing 11 actions:

```typescript
export class Plaid {
  // Link flow
  createLinkToken(ctx, { userId, products? })
  exchangePublicToken(ctx, { userId, publicToken, institutionId? })
  
  // Data retrieval  
  getAccounts(ctx, { userId, itemId? })
  getTransactions(ctx, { userId, accountId?, startDate?, endDate?, limit? })
  getLiabilities(ctx, { userId, itemId })
  getRecurringStreams(ctx, { userId, itemId })
  
  // Sync operations
  syncTransactions(ctx, { userId, itemId })
  syncLiabilities(ctx, { userId, itemId })
  refreshTransactions(ctx, { userId, itemId })
  
  // Enrichment
  enrichTransactions(ctx, { userId, transactionIds })
  
  // Orchestration
  runOnboarding(ctx, { userId, publicToken, products })
  
  // Webhooks
  registerWebhookRoutes(http, { path?, onWebhook? })
}
```

**Error handling** should use `ConvexError` with structured error codes:

```typescript
import { ConvexError } from "convex/values";

// In component actions
if (plaidError.error_code === "ITEM_LOGIN_REQUIRED") {
  throw new ConvexError({
    code: "PLAID_LOGIN_REQUIRED",
    itemId: args.itemId,
    message: "User must re-authenticate with their bank",
  });
}
```

## Conclusion

Converting your Plaid integration to a Convex component requires these key changes: moving to a class-based client that passes configuration as function arguments instead of reading `process.env`, restructuring auth to accept `userId` explicitly rather than using `ctx.auth`, exposing webhook handlers through a registration function rather than direct HTTP routes, and packaging with proper npm exports. The **Stripe component** (`@convex-dev/stripe`) serves as the closest architectural reference—examine its webhook registration pattern and class-based API design. Since no Plaid component exists in the ecosystem, this would fill a significant gap for Convex developers building fintech applications.
